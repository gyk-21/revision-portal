1️⃣ NGINX Architecture (Deep Technical)

Unlike traditional web servers like Apache (which typically handles one request per thread/process), NGINX is built from the ground up to use an asynchronous, non-blocking, event-driven architecture. This allows it to handle 10,000+ concurrent connections with a remarkably low memory footprint.

Master & Worker Processes

  • Master Process: Runs as root. Its primary role is to read and validate the configuration (nginx.conf), bind to privileged ports (like 80/443), and spawn/manage worker processes. It does not handle client requests directly.
  • Worker Processes: Run as an unprivileged user (e.g., nginx or www-data). They do the actual work of processing network connections, reading/writing to disk, and communicating with upstream servers. Typically, you configure one worker per CPU core (worker_processes auto;).

Event-Driven Model (epoll / kqueue)

Workers run a tight loop waiting for events on open sockets. Linux uses epoll and FreeBSD/macOS use kqueue. Instead of a context switch for every connection, a worker gets notified by the OS kernel only when a socket is ready for reading or writing. A single worker can concurrently manage thousands of active HTTP keep-alive connections.

Request Lifecycle Step-by-Step

  1. Client initiates a TCP handshake.
  2. Kernel alerts NGINX via epoll_wait() that a socket is ready.
  3. Worker Connection: NGINX worker accepts the connection and begins parsing the HTTP headers as bytes arrive.
  4. Request Processing: NGINX determines the correct server and location block.
  5. Upstream: If it's a proxy pass, the worker opens a non-blocking TCP socket to the upstream app server.
  6. Response: As the upstream sends the payload back, NGINX buffers it or streams it directly to the client socket via kernel-level zero-copy (e.g., sendfile).
NGINX vs Apache: Apache's MPM Prefork creates a heavy process per request, consuming vast memory at scale (C10k problem). NGINX's event loop multiplexes I/O, meaning 10k idle connections in NGINX cost only a few megabytes of RAM.

2️⃣ Installation & Configuration

Installation (RHEL / Ubuntu)

For production, always use the active stable or mainline repositories directly from nginx.org rather than OS-default repos to ensure the latest patches.

# Ubuntu / Debian
sudo apt update
sudo apt install curl gnupg2 ca-certificates lsb-release ubuntu-keyring
curl https://nginx.org/keys/nginx_signing.key | gpg --dearmor | sudo tee /usr/share/keyrings/nginx-archive-keyring.gpg >/dev/null
echo "deb [signed-by=/usr/share/keyrings/nginx-archive-keyring.gpg] http://nginx.org/packages/ubuntu `lsb_release -cs` nginx" | sudo tee /etc/apt/sources.list.d/nginx.list
sudo apt update && sudo apt install nginx

# RHEL / CentOS / AlmaLinux
sudo yum install epel-release
sudo yum install nginx
sudo systemctl enable --now nginx

Standard Directory Structure

  • /etc/nginx/nginx.conf: The main configuration file.
  • /etc/nginx/conf.d/: Directory for individual app/virtual host configs.
  • /var/log/nginx/: Default location for access.log and error.log.
  • /usr/share/nginx/html/: Default docroot for static files.

Context Hierarchy Breakdown

NGINX configs are structurally organized into cascading blocks called contexts.

user nginx;
worker_processes auto;          # MAIN CONTEXT (Global limits/processes)

events {
    worker_connections 1024;    # EVENTS CONTEXT (epoll, connections per worker)
}

http {                          # HTTP CONTEXT (Web traffic defaults)
    include /etc/nginx/mime.types;
    access_log /var/log/nginx/access.log;
    
    server {                    # SERVER CONTEXT (Virtual Host, Domain/IP binding)
        listen 80;
        server_name api.gyk.tech;
        
        location / {            # LOCATION CONTEXT (URI routing)
            proxy_pass http://localhost:8080;
        }
    }
}

3️⃣ Reverse Proxy & Load Balancing

As an architect, your main use for NGINX is likely as an entrance gateway. It offloads workloads, distributes traffic, and provides centralized logging.

Upstream Blocks & Algorithms

The upstream directive defines a cluster of servers to route logic to. NGINX supports several load-balancing mechanisms:

  • Round Robin (Default): Distributes requests evenly across the pool.
  • Least Connections: least_conn; routes traffic to the server with the fewest active TCP connections. Great for endpoints with variable latency.
  • IP Hash: ip_hash; maps the client's IPv4/IPv6 directly to a server. This implicitly provides session stickiness/persistence.
upstream backend_cluster {
    least_conn;                      # Algorithm selection
    server 10.0.1.10:8000 max_fails=3 fail_timeout=30s;
    server 10.0.1.11:8000 max_fails=3 fail_timeout=30s;
    server 10.0.1.12:8000 backup;    # Fallback disaster recovery node
}

server {
    listen 80;
    server_name myapp.com;

    location /api/ {
        proxy_pass http://backend_cluster;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        
        # Timeout Tuning
        proxy_connect_timeout 5s;
        proxy_send_timeout 15s;
        proxy_read_timeout 30s;
    }
}
Interview Q&A Focus: "How do you prevent NGINX from passing requests to a dead server?"
Answer: "Using max_fails and fail_timeout in the upstream block (Passive Health Checks) or NGINX Plus's health_check directive (Active Health Checks)."

4️⃣ SSL & Security

SSL termination at the NGINX layer offloads cryptographic decryption overhead from backend microservices.

TLS Tuning & Hardening

Architects must achieve high SSL Labs ratings by enforcing modern cipher suites and disabling legacy protocols like TLSv1.0/1.1.

server {
    listen 443 ssl http2;
    server_name secure.example.com;

    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # TLS Tuning
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;
    ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256...';
    
    # Session caching prevents fully renegotiating TLS on every request
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;

    # HSTS (Strict Transport Security)
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    
    # Security Headers
    add_header X-Frame-Options "SAMEORIGIN";
    add_header X-Content-Type-Options "nosniff";
}

Rate Limiting

Mitigate DDoS attacks and brute-force attempts by restricting RPS (Requests Per Second).

# Define zone in HTTP block: 10m bytes, rate 5 requests per second
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=5r/s;

server {
    location /login {
        # Allow a burst of 10 requests, delay others to adhere to rate
        limit_req zone=api_limit burst=10 nodelay;
    }
}

5️⃣ Performance Optimization

To tune NGINX for high concurrency, we configure limits at both the OS and NGINX level.

  • worker_processes auto: Spawns one worker per CPU core automatically.
  • worker_connections: Connections per worker. If auto gives 8 workers, and connections=1024, Max Clients = 8096.
  • ulimit -n (File descriptors): Must surpass worker_processes * worker_connections. Set worker_rlimit_nofile 100000; in main context.
  • keepalive: Keeps TCP connections between client/NGINX or NGINX/Upstream open to avoid 3-way handshake overhead.

Production Tuning Example

worker_processes auto;
worker_rlimit_nofile 100000;

events {
    worker_connections 4096;
    multi_accept on;
    use epoll;
}

http {
    # File I/O zero-copy
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;

    # GZIP Compression
    gzip on;
    gzip_comp_level 5;
    gzip_types text/plain application/json application/javascript text/css;

    # Buffering (Mitigate slowloris attacks)
    client_body_buffer_size 16K;
    client_header_buffer_size 1k;
    client_max_body_size 8m;
    large_client_header_buffers 4 8k;
}

6️⃣ Logging & Troubleshooting

Custom Log Formats

For ELK/Datadog ingestion, custom JSON logs are vastly superior to the default combined format. They provide metadata like upstream response times.

http {
    log_format apm_json escape=json 
    '{'
        '"time_local":"$time_local",'
        '"client_ip":"$remote_addr",'
        '"request":"$request",'
        '"status": "$status",'
        '"body_bytes_sent":"$body_bytes_sent",'
        '"request_time":"$request_time",'
        '"upstream_response_time":"$upstream_response_time",'
        '"upstream_addr":"$upstream_addr"'
    '}';

    access_log /var/log/nginx/access.json apm_json;
}

Incident Debugging: 502 Bad Gateway vs 504 Gateway Timeout

These are the two most common errors a DevOps engineer tackles.

  • 502 Bad Gateway: NGINX successfully reached out, but the upstream TCP port was closed, rejected the connection, or returned malformed data. Action: Check if app server container crashed.
  • 504 Gateway Timeout: NGINX established the connection, sent the payload, but the upstream application didn't respond within proxy_read_timeout. Action: Check database locks, expensive queries, or deadlocks in the backend app.

7️⃣ High Availability & Scalability

NGINX on a single compute instance is a SPOF (Single Point of Failure). Architecting for high availability requires an N-Tier setup.

Web Farm Architecture

  1. DNS Layer: Cloudflare or Route53 with Geo-routing.
  2. L4 Load Balancer: AWS ALB / HAProxy / Keepalived (VRIP) wrapping multiple NGINX instances.
  3. L7 Proxy Layer (NGINX): Multiple redundant NGINX instances doing SSL offloading, rate limiting, and HTTP routing.
  4. Compute Layer (Upstreams): EKS Pods, ECS Containers, or EC2 instances running the actual business logic.

Blue-Green Deployments

NGINX enables instant cut-over deployments by swapping upstreams.

# Deploy V2 to the backend_blue cluster
upstream backend {
    # server 10.0.1.50:80; # Version 1 (Green - drained)
    server 10.0.1.60:80; # Version 2 (Blue - active)
}
# Run `nginx -s reload` to switch traffic gracefully

8️⃣ DevOps & Cloud Integration

Docker Integration

Running NGINX in Docker implies daemon-less execution. The core command must block the container from exiting: CMD ["nginx", "-g", "daemon off;"].

Kubernetes Ingress (High-Level)

The ingress-nginx controller maps K8s Ingress objects dynamically into an nginx.conf ruleset across pods. It listens to the K8s API via Lua plugins and reloads dynamically without dropping active connections.

Zero-Downtime Reloads

A true Sysadmin knows how to apply new configurations without closing active TCP sessions.

# 1. Test configuration syntax FIRST
nginx -t

# 2. Issue a graceful reload 
# Master process parses new config, spawns new workers, 
# and tells old workers to shut down gracefully after finishing current requests.
nginx -s reload

9️⃣ Top 50 NGINX Interview Questions

1. What is NGINX?
An open-source, high-performance HTTP web server, reverse proxy, and API gateway driven by an asynchronous, event-driven architecture.
2. Main difference between NGINX and Apache?
Apache assigns a thread/process per connection (blocking). NGINX uses an event loop to handle thousands of connections per worker (non-blocking).
3. What is a "Master Process"?
The root process that reads configurations and maintains/spawns worker processes.
4. What is a "Worker Process"?
Unprivileged processes that use epoll/kqueue to handle actual network I/O and client requests.
5. How does `worker_processes auto;` work?
It detects the number of available CPU cores and spawns exactly one worker per core to minimize CPU context switching.
6. What is `epoll`?
A Linux kernel system call for scalable I/O event notification, the backbone of NGINX's concurrency.
7. What is the C10K problem?
The challenge of optimizing network sockets to handle 10,000 concurrent connections, which NGINX was explicitly built to solve.
8. How to test NGINX config without restarting?
Use the `nginx -t` command to validate syntax and paths.
9. How do you reload configs with zero downtime?
`nginx -s reload`. Master spawns new workers, old workers serve active connections then die gracefully.
10. Explain the Context hierarchy.
Main -> Events -> HTTP -> Server -> Location. Configuration directives cascade down.
11. What is a Reverse Proxy?
A server that sits in front of backend applications, routing client requests to them and shielding their direct identity.
12. What directive defines a reverse proxy?
`proxy_pass http://backend;`
13. What is an upstream block?
A context that defines a pool of backend servers for load balancing.
14. Default Load Balancing algorithm?
Round Robin. Requests are passed sequentially across the upstream pool.
15. When to use `least_conn`?
When request processing times vary significantly, ensuring new connections go to the least busy node.
16. How to achieve session stickiness?
Using the `ip_hash;` directive inside the upstream block.
17. Difference between Layer 4 and Layer 7 proxying?
L4 uses the `stream` context (TCP/UDP, raw IPs). L7 uses the `http` context (HTTP headers, URIs).
18. Fix for generic 502 Bad Gateway?
Check if the upstream application service is running, listening on the right port, or restricted by firewalls.
19. Fix for 504 Gateway Timeout?
Increase `proxy_read_timeout` if the backend needs more time, or debug database queries dragging down the backend.
20. What is `X-Forwarded-For`?
An HTTP header injected by NGINX to pass the original client's public IP address to the backend application.
21. How to enable SSL?
`listen 443 ssl; ssl_certificate /path.crt; ssl_certificate_key /path.key;`
22. What is SSL Termination?
Decrypting HTTPS traffic at the NGINX layer and sending plain HTTP to the internal backend networks, saving app CPU.
23. How to enforce HTTPS only?
In port 80 server block, add `return 301 https://$host$request_uri;`.
24. What is HSTS?
Strict Transport Security - a header informing browsers to structurally deny HTTP connections to this domain in the future.
25. How do you limit brute force attacks?
Using `limit_req_zone` and `limit_req` directives to throttle requests per second per IP.
26. Command to drop connections outright?
`return 444;` closes the connection instantly without returning an HTTP header.
27. How to restrict access by IP?
`allow 192.168.1.0/24; deny all;` inside a location or server block.
28. What is `server_tokens off;`?
A security directive that removes the NGINX version number from the Server response header and error pages.
29. How to configure Basic Auth?
`auth_basic "Restricted"; auth_basic_user_file /etc/nginx/.htpasswd;`
30. Can NGINX block SQL Injection directly?
Out of the box, no. It requires NAXSI or ModSecurity (WAF modules) integration.
31. What is `worker_connections`?
The maximum number of simultaneous connections that a single worker process can handle.
32. How to calculate max total connections?
`worker_processes * worker_connections` (Accounting for file descriptor OS limits).
33. Why use `sendfile on;`?
It allows data transfer directly from disk to TCP socket within kernel space, avoiding costly context switches to user space.
34. What is `tcp_nopush`?
Optimizes `sendfile` by forcing NGINX to send HTTP response headers in the same packet as the file data chunk.
35. Purpose of HTTP Keep-Alive?
Allows multiple requests/responses over a single TCP connection, drastically reducing 3-way handshake overhead.
36. What does `gzip on;` do?
Compresses text payloads (HTML, CSS, JSON) on the fly, reducing bandwidth usage at the cost of slight CPU overhead.
37. How to handle "Worker connections are not enough"?
Increase `worker_connections`, but remember to also increase the OS file descriptor limit (`ulimit -n` and `worker_rlimit_nofile`).
38. What is Microcaching?
Caching dynamic content (like API responses) for extremely short intervals (1-2 seconds) using `proxy_cache`. Handles traffic spikes effectively.
39. FastCGI vs Proxy Pass?
`fastcgi_pass` speaks the FastCGI binary protocol (often used with PHP-FPM). `proxy_pass` speaks HTTP.
40. Where do you tune large file upload limits?
`client_max_body_size 50M;` to prevent HTTP 413 Payload Too Large errors.
41. File paths for default logs?
`/var/log/nginx/access.log` and `/var/log/nginx/error.log`.
42. How to log in JSON format?
Define a `log_format` payload wrapping variables in JSON brackets, and apply it to `access_log /path format_name;`.
43. What is `$request_time`?
Total time (in seconds with ms resolution) from reading the first client bytes to sending the last bytes.
44. What is `$upstream_response_time`?
Time it took precisely for the backend upstream pool to respond. Crucial for tracing latency.
45. How to enable debug logging?
Recompile NGINX with `--with-debug` (or use debug binary) and append `debug` to error_log: `error_log /map.log debug;`.
46. Role in Kubernetes?
Used universally as the default Ingress Controller (`ingress-nginx`) mapping internet traffic to ClusterIP services.
47. Why `daemon off` in Docker?
Containers exit if their primary PID (PID 1) backgrounds itself. `daemon off` keeps NGINX in the foreground.
48. What is location modifier `~*`?
It signifies a case-insensitive Regular Expression match for URIs.
49. Priority of Location blocks?
Exact match (`=`), Preferential Prefix (`^~`), Regex (`~`, `~*`), then Standard Prefix (`/`).
50. If two upstreams are equal, how does RR choose?
Round Robin strictly alternates sequentially unless weighted (`weight=3`).
