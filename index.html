SECTION 01

IIS Architecture

1.1 Component-Level Architecture

IIS (Internet Information Services) is a modular, extensible web server built into Windows Server. Its architecture separates concerns across kernel-mode and user-mode layers, providing performance, stability, and isolation.

LIVE ARCHITECTURE — Request Flow Animation

Component Descriptions

ComponentLayerRole & Details
HTTP.sysKernel ModeWindows kernel-mode HTTP stack. Handles all TCP connections, SSL/TLS offloading, URL-based routing, and kernel-mode response caching. Introduced in Windows Server 2003. A crash in user-mode w3wp.exe does NOT take down this layer.
W3SVC (World Wide Web Svc)User Mode (svchost)Manages HTTP.sys URL reservations and is the management interface for IIS configuration changes. Service must be running for IIS HTTP functionality. Interface for appcmd, IIS Manager, and PowerShell WebAdministration.
WAS (Windows Activation Svc)User Mode (svchost)Split from W3SVC in IIS 7+. Reads ApplicationHost.config. Manages App Pool lifecycle — starts, stops, and recycles worker processes (w3wp.exe). Also handles non-HTTP protocols: net.tcp, net.pipe, net.msmq.
Application PoolUser ModeIsolation boundary in IIS. Each app pool has its own w3wp.exe with its own memory space, identity, CLR version, and recycling settings. Provides fault isolation — a crash in one pool does not affect others.
w3wp.exe (Worker Process)User ModeThe actual HTTP processing unit. Hosts the ASP.NET/ISAPI pipeline. Runs in user mode. One per app pool (or multiple in Web Garden). All your application code runs inside this process.
ApplicationHost.configConfig LayerMaster config file at %SystemRoot%\System32\inetsrv\config\. Stores all IIS settings: sites, bindings, app pools, modules, handlers. The single source of truth for IIS configuration. WAS reads this file.
Web.configApp ConfigApplication-level config. Inherits from ApplicationHost.config via the config hierarchy. Developers use this for app-specific settings, handlers, modules, authentication rules, and connection strings.

1.2 Request Flow — Animated Step-by-Step Walk-Through

Each step below represents a stage in the request journey through IIS. Watch the architecture diagram above for the animated packet traversal.

①
Browser Initiates TCP Connection

Browser resolves DNS → connects to server IP on port 80 (HTTP) or 443 (HTTPS). TLS handshake occurs for HTTPS — CPU-intensive step handled by HTTP.sys in kernel mode.

▼
②
HTTP.sys Receives Request (Kernel Mode)

HTTP.sys driver intercepts the incoming HTTP request. Checks kernel-mode response cache. If cached → returns directly without waking user mode. If not cached → passes to W3SVC/WAS.

▼
③
W3SVC Routes to WAS

W3SVC receives request metadata. Identifies which Application Pool should handle it based on URL and site bindings configured in ApplicationHost.config.

▼
④
WAS Starts/Selects Worker Process

WAS checks if a worker process (w3wp.exe) is already running for the target App Pool. If not → WAS spawns a new w3wp.exe. If yes → routes to the existing process.

▼
⑤
w3wp.exe HTTP Pipeline Execution

The worker process executes the integrated pipeline: Authentication → Authorization → URL routing → Handler mapping → Module execution → Your application code runs here.

▼
⑥
Application Code Executes

ASP.NET/ISAPI/PHP handler processes the request. Hits databases, caches, downstream services. Builds the HTTP response (HTML, JSON, file stream, etc.).

▼
⑦
Response Travels Back Through Pipeline

Response flows back through remaining pipeline events: UpdateCache → LogRequest → EndRequest. Compression modules, custom headers, and output filters applied here.

▼
⑧
HTTP.sys Returns Response to Browser

HTTP.sys sends the response back over the TCP connection. Optionally caches in kernel mode for future identical requests. Connection kept alive or closed per Keep-Alive header.

🧠 MEMORY TRICK"HTTP-WAS-POOL-PIPE-APP" → HTTP.sys → WAS → App Pool → Pipeline → App Code → Response. Say it 3 times before the interview!

3.2 App Pool Recycling

Recycling terminates the current w3wp.exe and starts a fresh one. This prevents memory bloat from leaks and keeps the application healthy. With overlapped recycling (default ON), new w3wp starts before old one stops — near-zero downtime.

Recycling TypeDescription & Production Guidance
Time-basedRecycle every N minutes (default: 1740 mins = 29 hrs). Production tip: Disable time-based recycling (set to 0) — unpredictable timing can align with peak traffic.
Request CountRecycle after N requests handled. Good for leak-prone apps where memory grows proportionally with requests processed.
Memory (Private Bytes)Trigger recycle when private memory exceeds threshold (e.g., 512MB). Preferred production strategy — react to actual memory pressure.
Schedule-basedRecycle at specific times (e.g., 3 AM daily). Predictable but can cause issues if off-peak has maintenance jobs or batch processing.
Overlapped RecyclingDEFAULT ON. New w3wp starts BEFORE old one stops. Old process drains existing requests while new one handles incoming. Enables zero-downtime recycling. Always keep enabled.
Event Log on RecycleEvent IDs: 5074 (recycle scheduled), 5075 (recycle condition hit). Monitor in Event Viewer → System log. Alert if recycling too frequently (memory leak indicator).
# ─── Configure App Pool recycling via PowerShell ────────────────────────────── $appPool = "MyAppPool" # Disable time-based recycling (set to 0 = never) Set-ItemProperty IIS:\AppPools\$appPool Recycling.periodicRestart.time "00:00:00" # Set private memory limit (KB) – 512MB Set-ItemProperty IIS:\AppPools\$appPool Recycling.periodicRestart.privateMemory 524288 # Set specific recycle schedule time (3 AM) $recycleTime = [TimeSpan]"03:00:00" Add-WebConfigurationProperty -Filter "system.applicationHost/applicationPools/add[@name='$appPool']/recycling/periodicRestart/schedule" -Name "." -Value @{value=$recycleTime} # View current recycle settings Get-ItemProperty IIS:\AppPools\$appPool | Select -ExpandProperty Recycling # Manually recycle pool (uses overlapped recycling) Restart-WebAppPool -Name $appPool

4.3 Security Hardening Checklist

Hardening ActionDetails & Implementation
Remove IIS Version HeaderSet <requestFiltering removeServerHeader="true"> or use UrlScan. Removes "Server: Microsoft-IIS/10.0" header. Prevents version disclosure.
Disable Directory BrowsingIIS Manager → Site → Directory Browsing → Disable. Never expose directory listings — attackers can enumerate all files in web root.
Request FilteringBlock dangerous extensions (.config, .bak, .log, .cs), limit request size (maxAllowedContentLength), block double-encoding (%25), limit URL length.
Disable Unused HTTP VerbsAllow only GET, POST, HEAD. Block TRACE (XST attack), OPTIONS, PUT, DELETE unless specifically needed by application design.
Custom Error PagesReplace default IIS error pages. Prevent stack traces and server version information leakage. Use generic custom pages that log internally.
X-Frame-Options HeaderAdd SAMEORIGIN or DENY to prevent clickjacking attacks. Add via custom headers in web.config <httpProtocol><customHeaders>.
Content-Security-PolicyDefine allowed script, style, font sources. Prevent XSS. Add as custom response header. Start with report-only mode to detect violations first.
Require HTTPSSSL Settings → Require SSL. Enforce in URL Rewrite (permanent redirect). Add HSTS header. Submit domain to HSTS preload list for maximum protection.
Least Privilege IdentityUse ApplicationPoolIdentity. Grant only needed NTFS permissions to content directory (Read/Execute for site, Modify for upload/log directories).
Patch & UpdateKeep Windows Server and IIS patched. Monitor security advisories at microsoft.com/security. Subscribe to MSRC security update emails.
WAF / ARR ProtectionPlace Application Request Routing or WAF (F5, Imperva, Azure WAF) in front of IIS. Provides OWASP rule-based protection for SQL injection, XSS, etc.
Audit LoggingEnable Windows Security audit logging. Monitor Event Viewer for 4625 (failed logon), 4648 (explicit logon). Forward to SIEM for correlation.
⚠️ CRITICAL VULNERABILITIES TO KNOW FOR INTERVIEWSCVE-2017-7269 (IIS 6 WebDAV buffer overflow — remote code execution), CVE-2015-1635 (HTTP.sys remote code execution "MS15-034"), CVE-2021-31166 (HTTP.sys denial of service). Always mention keeping Windows Update current and checking MSRC bulletins for IIS-specific CVEs.

5.4 Real-World Production Troubleshooting Scenarios

🔴 Scenario 1: Application returning 503 Service Unavailable
  1. Check IIS Manager → Application Pools → Is the pool Stopped? Start it manually.
  2. If it immediately stops again → Rapid Fail Protection triggered (5 failures in 5 min).
  3. Check Event Viewer → System → Look for Event ID 5117 (worker process failure) or 5059 (failure to start).
  4. Check HTTPERR log at %SystemRoot%\System32\LogFiles\HTTPERR\ for "Connection_Refused" or "AppOffline".
  5. Check for app_offline.htm file in site root — if present, IIS returns 503 intentionally.
  6. Check disk space on C: drive — IIS logs or temp files may have filled disk causing failures.
  7. Restart App Pool: Restart-WebAppPool -Name "PoolName"
  8. If recurring: Enable FREB, check application error details, look for memory or CPU exhaustion.
🟠 Scenario 2: Intermittent 500 Errors in Production
  1. Enable FREB for status code 500 on the production site immediately.
  2. Enable detailed errors for localhost only: <httpErrors existingResponse="PassThrough" errorMode="DetailedLocalOnly" />
  3. Use Remote Desktop → Browse site locally to see full error details without exposing to all clients.
  4. Check Application Event Log for unhandled .NET exceptions and their stack traces.
  5. Review IIS access logs for time-taken values — slow responses indicate database or timeout issues.
  6. Check .NET CLR exceptions in Performance Monitor: .NET CLR Exceptions → # of Excepts Thrown/sec.
  7. If 500.19: Validate web.config XML syntax — use: appcmd.exe validate config "SiteName"
🟡 Scenario 3: High Memory Usage / w3wp.exe Memory Leak
  1. Confirm leak: Task Manager → Process tab → Find w3wp.exe — is Private Bytes growing continuously over time?
  2. Identify which app pool: tasklist /fi "imagename eq w3wp.exe" /fo list — note PID, then match to IIS pool.
  3. Take process memory dumps at intervals: procdump -ma {PID} C:\dumps\ (download Sysinternals procdump).
  4. Analyze dumps in WinDbg: !dumpheap -stat to find object counts, !gcroot {address} to find what holds references.
  5. Look for: static collections growing unboundedly, event subscriptions not unsubscribed, undisposed HttpClient instances.
  6. Short-term fix: Configure private memory recycling threshold (e.g., 512MB) in App Pool settings.
  7. Long-term fix: Profile application with dotMemory, ANTS Memory Profiler, or Visual Studio Diagnostic Tools.
🔵 Scenario 4: Authentication Failures (401 Errors)
  1. Check FREB sub-status code: 401.1 (bad credentials), 401.2 (login format), 401.3 (NTFS ACL issue).
  2. For Windows Auth: Check SPN registration — setspn -L {computername} — missing SPN causes NTLM fallback or failure.
  3. Test Kerberos vs NTLM: Use Fiddler to inspect WWW-Authenticate header in the 401 challenge response.
  4. Double-hop issue (Server A → Server B fails): Configure constrained delegation in Active Directory or use CredSSP.
  5. Check if browsers send credentials automatically: Must be in Intranet Zone for automatic Windows Auth negotiation.
  6. For Basic Auth: Verify HTTPS is enforced — never allow Basic Auth over plain HTTP in production.

7.1 Application Request Routing (ARR) Architecture

ARR Configuration Key Points

FeatureDetails
Server FarmLogical group of backend web servers. Created in IIS Manager → Server Farms node after ARR module installation.
Health MonitoringARR polls backends via HTTP probe. Configurable: URL path, interval (seconds), timeout, expected status code. Removes unhealthy node from rotation automatically.
ARR Affinity CookieARRAffinity / ARRAffinitySameSite cookie. Routes returning clients to same backend. Enable for stateful apps with in-memory state. Disable for stateless apps.
